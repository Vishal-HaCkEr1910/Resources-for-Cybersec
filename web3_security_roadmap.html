SimpleStorage.sol
// SPDX-License-Identifier: MIT pragma solidity ^0.8.18; contract SimpleStorage { // Storage slot 0 — costs 20,000 gas to write for first time uint256 public favoriteNumber; // Structs pack data together struct Person { string name; // dynamic — stored as hash uint256 favoriteNum; // fixed — 32 bytes } // Dynamic array — length at slot, items at keccak256(slot)+i Person[] public people; // Mapping — items at keccak256(key . slot) mapping(address => uint256) public addrToNum; mapping(string => uint256) public nameToNum; function store(uint256 _number) public { favoriteNumber = _number; // SSTORE — 5,000 gas (warm) or 20,000 (cold) } // 'view' means read-only — no gas when called externally function retrieve() public view returns (uint256) { return favoriteNumber; // SLOAD } // 'memory' = temporary, lives only during this function call function addPerson(string memory _name, uint256 _num) public { people.push(Person(_name, _num)); addrToNum[msg.sender] = _num; nameToNum[_name] = _num; } }
2
Deploy and test every function. Add 5 people. Retrieve by index. Check mappings for your own address.
Then: Change memory to storage in addPerson and try to compile. Read the error carefully.
What is the difference between storage, memory, and calldata? When does the EVM use SSTORE vs MSTORE? Why does storage cost more gas? Write this in plain English as if explaining to a friend who codes in C.

// SPDX-License-Identifier: MIT pragma solidity ^0.8.18; contract FundMe { address public immutable owner; // immutable = cheaper than storage mapping(address => uint256) public addressToAmount; address[] public funders; constructor() { owner = msg.sender; // set at deploy time } // AUDIT POINT: this modifier pattern is used everywhere modifier onlyOwner() { require(msg.sender == owner, "FundMe: Not owner"); _; // rest of function executes here } // payable = can receive ETH function fund() public payable { require(msg.value > 0, "FundMe: Must send ETH"); addressToAmount[msg.sender] += msg.value; funders.push(msg.sender); } function withdraw() public onlyOwner { // Reset all funders for (uint256 i = 0; i < funders.length; i++) { addressToAmount[funders[i]] = 0; } funders = new address[](0); // Best practice: use call() not transfer() or send() (bool success, ) = payable(owner).call{value: address(this).balance}(""); require(success, "FundMe: Transfer failed"); } receive() external payable { fund(); } fallback() external payable { fund(); } }
2

• Fund from 3 different accounts (switch accounts in Remix)
• Try to fund with 0 ETH — what happens?
• Try to withdraw from a non-owner account — what error appears?
• Withdraw from the owner account — verify the balance moved correctly

⚙️ Practice — 2 Hours
1
Build and test a minimal proxy pattern — the foundation of every upgradeable contract:
Proxy.sol — Storage Collision Demo
// SPDX-License-Identifier: MIT pragma solidity ^0.8.18; contract Logic { uint256 public value; // slot 0 in Logic address public sender; // slot 1 in Logic function setValue(uint256 _val) public { value = _val; // writes to slot 0 sender = msg.sender; // writes to slot 1 } } contract Proxy { // CRITICAL: must match Logic's slot layout EXACTLY uint256 public value; // slot 0 — same as Logic.value address public sender; // slot 1 — same as Logic.sender address public logicAddress; // slot 2 — proxy admin constructor(address _logic) { logicAddress = _logic; } fallback() external payable { // delegatecall: run Logic's code BUT in THIS contract's storage (bool success, ) = logicAddress.delegatecall(msg.data); require(success, "delegatecall failed"); } }
2
Test sequence:
① Deploy Logic → copy its address
② Deploy Proxy with Logic's address
③ In Remix: at the contract selector, use Proxy's address but Logic's ABI (copy paste the setValue function interface)
④ Call setValue(42) on Proxy
⑤ Read value on Proxy — should be 42
⑥ Read sender on Proxy — should be YOUR address (not Logic's address!)
⑦ Read value on Logic — should be 0 (untouched!)
In your own words: when you call a function on the Proxy with delegatecall, who is msg.sender? Whose storage gets modified? What happens if Logic has a variable at slot 0 but Proxy has a different variable at slot 0? This is called "storage collision" — it's a critical vulnerability class you'll exploit in Ethernaut.

Events and ABI Practice
pragma solidity ^0.8.18; contract EventsAndABI { // Custom errors (cheaper than string require) error NotOwner(address caller, address required); error InsufficientBalance(uint256 requested, uint256 available); // 'indexed' topics = searchable in logs (max 3 per event) event Transfer(address indexed from, address indexed to, uint256 amount); address public owner; mapping(address => uint256) public balances; constructor() { owner = msg.sender; } function deposit() public payable { balances[msg.sender] += msg.value; emit Transfer(address(0), msg.sender, msg.value); } function transfer(address _to, uint256 _amount) public { if (balances[msg.sender] < _amount) revert InsufficientBalance(_amount, balances[msg.sender]); balances[msg.sender] -= _amount; balances[_to] += _amount; emit Transfer(msg.sender, _to, _amount); } // Compute any function's selector — critical for calldata analysis function getSelector(string memory _sig) public pure returns (bytes4) { return bytes4(keccak256(bytes(_sig))); } }
1
Call getSelector("transfer(address,uint256)") — it should return 0xa9059cbb. This is the ERC20 transfer selector. Every Ethereum transaction starts with these 4 bytes if it's calling transfer.
Can you decode the first 4 bytes of any transaction on Etherscan? Go to etherscan.io, find any recent ERC20 transfer transaction, and verify the function selector matches what you computed.

MyToken.sol — Pure ERC20 Implementation
pragma solidity ^0.8.18; contract MyToken { string public name; string public symbol; uint8 public decimals = 18; uint256 public totalSupply; mapping(address => uint256) public balanceOf; mapping(address => mapping(address => uint256)) public allowance; event Transfer(address indexed from, address indexed to, uint256 value); event Approval(address indexed owner, address indexed spender, uint256 value); constructor(string memory _name, string memory _symbol, uint256 _supply) { name = _name; symbol = _symbol; totalSupply = _supply * 10**18; balanceOf[msg.sender] = totalSupply; emit Transfer(address(0), msg.sender, totalSupply); } function transfer(address _to, uint256 _value) public returns (bool) { require(balanceOf[msg.sender] >= _value, "Insufficient balance"); balanceOf[msg.sender] -= _value; balanceOf[_to] += _value; emit Transfer(msg.sender, _to, _value); return true; } function approve(address _spender, uint256 _value) public returns (bool) { allowance[msg.sender][_spender] = _value; emit Approval(msg.sender, _spender, _value); return true; } function transferFrom(address _from, address _to, uint256 _value) public returns (bool) { require(balanceOf[_from] >= _value, "Insufficient balance"); require(allowance[_from][msg.sender] >= _value, "Insufficient allowance"); balanceOf[_from] -= _value; balanceOf[_to] += _value; allowance[_from][msg.sender] -= _value; emit Transfer(_from, _to, _value); return true; } }
What is the ERC20 approval race condition? Look up "ERC20 approve frontrun attack". Why does USDC use increaseAllowance/decreaseAllowance instead of plain approve? This is a real vulnerability you'll see in audits.

📅 The 8-Day Shadow Audit Protocol
Day 181
Architecture Review (4 hrs): Clone the repo. Read all Solidity files top to bottom. Draw the architecture: which contracts call which. Identify the entry points. Understand the protocol's purpose.
Day 182
Automated Analysis (4 hrs): Run Slither. Run your checklist. Triage Slither output — classify every finding. Note suspicious areas for manual review.
Day 183
Manual Review — Access Control (4 hrs): Go through every public/external function. Who can call it? Should they be able to? Check every modifier. Check every owner/admin pattern.
Day 184
Manual Review — State & Math (4 hrs): Every state update — is it in the right order (checks-effects-interactions)? Every arithmetic — could it overflow? Every division — could precision be lost?
Day 185
Foundry Tests + Fuzz (4 hrs): Write tests for suspicious areas. Write fuzz tests for any function that takes user input. Run with 10,000+ iterations.
Day 186
Write Your Report (4 hrs): For every finding, fill in this template:

## [H-01] Title of finding
**Severity:** High/Med/Low
**Description:** What is the bug?
**Impact:** What can attacker do?
**Proof of Concept:** (Foundry test)
**Recommendation:** How to fix?
Day 187
Compare with Official Report (4 hrs): Read the official C4 report. Compare finding by finding. For every High/Critical you missed — read the code again. Understand exactly why you missed it. Add the pattern to your checklist.
Day 188
Score Yourself + Document: Highs caught / Total Highs = your detection rate. Target: >50% before going live. Save your report to GitHub. This is your portfolio.
